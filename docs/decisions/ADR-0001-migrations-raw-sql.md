# 0001. Use numbered raw SQL migrations

Date: 2026-04-20

## Status

Accepted

## Context

This service needed a migration pattern that was:

- Readable without tooling — a new engineer should be able to `cat` the
  file and understand the schema change.
- Runnable from any environment with `psql` — no Python import graph
  required during deploy.
- Auditable — migration number + filename is stable git history; no
  migration hash churn from autogenerated scripts.

Alembic was considered and rejected because autogenerated migrations
are frequently incorrect for Postgres-specific features (TEXT[], partial
indexes, check constraints) that this schema uses, and manual cleanup
of autogenerated files creates a worse audit trail than starting
manually.

## Decision

Migrations live in `migrations/NNN_description.sql` where `NNN` is a
zero-padded sequence number starting at `001`. Each file is idempotent
Postgres SQL — `CREATE TABLE IF NOT EXISTS`, `ALTER TABLE … IF NOT
EXISTS`, etc. Migrations are applied out-of-band via Railway's deploy
hook (configured on the service, not in ci.yml) so they run exactly
once per deploy.

## Consequences

**Easier:**
- Reviewing migration contents in PRs — plain SQL, no Python to trace.
- Running migrations locally — `psql -f migrations/NNN_*.sql`.
- Rollbacks when needed — write `migrations/NNN_revert_X.sql` as a
  first-class forward migration.

**Harder:**
- Schema introspection from Python — SQLAlchemy models must be kept
  in sync with migration DDL by hand.
- Migration state tracking — no built-in `alembic_version` table; we
  rely on Railway's deploy hook running migrations in order and on
  migrations being idempotent.

**Cost of this decision against standards:**
- API-011 flags absence of `alembic upgrade head` in ci.yml. Exempted
  in evaluator.yaml with this ADR as the rationale.
